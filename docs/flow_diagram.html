ALL GROQ/LLM CALLS — WHAT FLOWS IN AND OUT
① EndpointClassifier (Phase ③b)
IN: endpoint list + tech stack
OUT: attack surface classification per endpoint (auth/upload/admin/api/…)
② PayloadGenerator (Phase ④)
IN: endpoint type, method, params, tech context
OUT: custom payloads (SQLi/XSS/SSTI/etc) targeted to endpoint
③ PayloadMutator (Phase ④)
IN: baseline payloads + WAF fingerprint
OUT: obfuscated/mutated payload variants for WAF evasion
④ reason_chain_from_vulns (Phase ⑤)
IN: confirmed_vulns[], tech_stack[], wp_users[], target — max_tokens=2000
OUT: JSON chains[] with steps[{produces[], consumes[], depends_on[]}], preconditions_missing[]
⑤ ExploitSelector (Phase ⑤)
IN: all chains + vulns + tech + signals + primitives
OUT: SelectedStrategy {chain_name, confidence, reasoning}
⑥ GapAnalyzer (Phase ⑤b, ×4 iterations)
IN: missing conditions, tech stack, fulfilled conditions, step_produced_data (creds/tokens)
OUT: {condition, module, rationale, target_scope} — dispatches probe module
⑦ LLMAnalyzer (Phase ⑥b)
IN: all vulns, findings, endpoints, chains, tech stack
OUT: risk score, attack path suggestions, remediation advice
KNOWN ARCHITECTURAL PROBLEMS (technical debt)
⚠ Hardcoded fallback chains
_run_chain_planning_phase() has a hardcoded xmlrpc_bruteforce chain injected when Groq returns 0 chains. This bypasses all data-flow reasoning.
✓ FIXED — Scanner↔findings sync
dalfox_runner._parse_findings(): now recognises [POC][V/R][METHOD][context] format. [V] = verified=True, confidence=0.95. [R] = unverified, confidence=0.8.
_promote_dalfox_result(): sets verified=has_verified_poc in the vuln object.
_promote_sqlmap_result(): sets verified=True unconditionally when vulnerable=True.
_run_xss_phase() / _run_boolean_sqli_phase(): now merges custom detector results with existing dalfox/sqlmap findings instead of overwriting them.
⚠ GapAnalyzer tool calls are advisory only
Groq recommends a module, but the actual tool call is hardcoded in gap_analyzer.py dispatch map. Groq cannot call arbitrary tools — it only selects from a fixed list of 8 modules.
⚠ 60+ phases in agent.py
Many phases (_run_zero_day_phase, _run_container_escape_phase, _run_log_evasion_phase, etc.) are stubs or near-stubs. Creates noise without value in most scans.
⚠ combine_chains() never called
ChainPlanner.combine_chains() exists but is not wired into the main flow. Groq-generated chains are used directly without being merged/combined.
⚠ max_tokens=500 in all Groq calls
Fixed — reason_chain_from_vulns now passes max_tokens=2000. But other Groq calls (PayloadGenerator, EndpointClassifier, LLMAnalyzer) still default to 500, which may truncate complex responses.
